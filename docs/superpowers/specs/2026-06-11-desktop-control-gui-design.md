# SmartFinger Desktop Control GUI Design

## Goal

为 `SmartFinger` 增加一个仅在电脑本地使用的控制页面，替代旋钮作为主要操作入口。用户通过浏览器直接连接 ESP32 串口，在页面中完成档位切换、频率控制、立即启动、倒计时启动、指定时刻启动和运行状态查看。电脑当前时间自动同步给设备，用户不再需要手工输入“当前时间”。

## Scope

本次设计只覆盖本地浏览器控制页和设备串口协议扩展，不做桌面 App 打包，不做手机端页面，不做 BLE。

包含：

1. 本地网页 GUI。
2. 浏览器通过 Web Serial 连接 ESP32。
3. 自动发送电脑当前时间给设备。
4. 档位 `0-6` 的直接控制。
5. 频率参数控制，至少支持预设档位和自定义输入。
6. 立即开始、立即停止、取消定时。
7. 倒计时启动。
8. 指定目标时刻启动。
9. 串口状态、设备状态、日志回显。

不包含：

1. 原生桌面应用打包。
2. 手机浏览器支持。
3. 云同步或远程控制。
4. 复杂权限系统。

## User Problem

当前流程主要依赖旋钮和串口文本命令。对时操作费劲，用户需要在关键时刻尽量快地完成以下动作：

1. 连接设备。
2. 看到电脑当前时间。
3. 设定目标开始时刻或倒计时。
4. 选择档位和频率。
5. 在等待期间确认设备是否已武装。
6. 在运行后确认 CPS、剩余时间和日志输出。

串口手输命令虽然可行，但对临场操作不友好。GUI 的目的不是替代固件核心逻辑，而是让这些操作进入“看得见、点得快、状态清楚”的界面。

## Recommended Approach

采用单页本地控制台。

原因：

1. 所有关键操作都需要快速往返切换，单页比多步骤向导更适合临场使用。
2. 当前系统本身已经有串口协议基础，在浏览器里直接发命令比重写一套复杂通信层更稳。
3. 单页结构最容易同时展示当前时间、目标时间、倒计时和设备运行状态。

不采用分步向导，因为它会增加切换成本。

不采用原生桌面打包，因为当前阶段收益不高，Web Serial 足够满足“电脑本地使用”。

## Architecture

系统分成两部分：

### 1. `firmware/` 固件

继续作为唯一设备逻辑源。

职责：

1. 接收串口命令。
2. 更新档位和运行参数。
3. 进入等待武装状态或立即启动。
4. 执行 `Grade 6` 的 5 秒高仿真点击流程。
5. 持续回传状态和日志。

### 2. `desktop-control/` 本地控制页

一个静态前端页面，通过浏览器 Web Serial API 与板子通信。

职责：

1. 连接串口。
2. 自动取电脑系统时间。
3. 点击按钮后自动发送 `TIME`、`AT`、`IN`、`CANCEL`、`STATUS` 等命令。
4. 根据串口回显刷新界面状态。
5. 提供档位和频率的直接控制。

## UI Structure

单页控制台分为四块：

### 1. 连接与时间

显示：

1. 串口连接按钮。
2. 当前串口名。
3. 电脑当前时间。
4. 设备连接状态。

行为：

1. 用户点击连接后选择 `/dev/cu.usbmodem...`。
2. 连接成功后页面自动开始周期性发送当前时间同步命令。

### 2. 档位与频率

显示：

1. `Grade 0-6` 按钮组。
2. 当前选中档位。
3. 自定义频率相关输入。

行为：

1. 用户点击任意档位时，页面直接发送对应控制命令。
2. 用户可以切换到预设档位，也可以调自定义频率。
3. 旋钮不再作为主要控制入口；固件仍可兼容存在的旋钮硬件，但 GUI 是主入口。

### 3. 启动控制

显示：

1. 立即开始按钮。
2. 停止按钮。
3. 倒计时输入框。
4. 指定开始时间输入框。
5. 设定按钮。
6. 取消定时按钮。

行为：

1. 倒计时启动时，页面直接以电脑当前时间计算并发送命令。
2. 指定时刻启动时，页面先同步当前时间，再发目标时刻命令。
3. 等待期间设备进入 `ARMED` 状态。

### 4. 状态与日志

显示：

1. 当前模式。
2. 当前步骤，如 `WAIT`、`CALC`、`PRESS`、`RELEASE`、`DONE`。
3. 剩余倒计时。
4. 最近一次运行的 CPS。
5. 串口日志滚动区。

行为：

1. 所有串口回显都进入日志区。
2. GUI 从日志中提取关键状态并更新界面。

## Time Sync Design

当前时间不由用户手填，而由电脑自动同步。

规则：

1. 浏览器每次建立串口连接后立刻发送一次当前时间。
2. 用户设置“指定开始时间”前，再补发一次当前时间，减少漂移。
3. 如果用户等待时间较长，页面可按固定间隔再次同步当前时间。
4. 同步失败时，界面需要明确提醒“设备时钟未同步”。

这样用户只需要关心“目标时间是什么”，不需要再手工填“现在几点”。

## Device Protocol Changes

现有命令保留并扩展。

保留：

1. `TIME HH:MM:SS`
2. `AT HH:MM:SS`
3. `IN <seconds>`
4. `CANCEL`
5. `STATUS`

新增建议：

1. `GRADE <0-6>`
   直接设置当前档位。
2. `START`
   立即启动当前档位，尤其用于第 6 档。
3. `STOP`
   立即停止。
4. `FREQ <preset|custom params>`
   更新当前频率参数。
5. 结构化状态回显，如：
   `STATE mode=ARMED grade=6 left_ms=4200 step=WAIT`

结构化回显的目的，是让 GUI 不用只靠人类可读日志字符串去猜状态。

## State Model

GUI 需要识别这些高层状态：

1. `DISCONNECTED`
2. `CONNECTED_IDLE`
3. `ARMED`
4. `RUNNING`
5. `COMPLETED`
6. `ERROR`

设备侧保留更细的 `AdaptiveStep`：

1. `WAIT`
2. `CALC`
3. `PRESS`
4. `RELEASE`
5. `DONE`

GUI 只需要把它们映射到清晰可见的标签。

## Error Handling

需要覆盖以下异常：

1. 用户浏览器不支持 Web Serial。
   页面显示明确提示，不隐藏问题。
2. 串口断开。
   进入 `DISCONNECTED`，按钮禁用。
3. 当前时间尚未同步就尝试 `AT`。
   页面先自动同步；若失败，阻止提交。
4. 设备回传异常或格式不对。
   日志区保留原始文本，同时界面显示“状态解析失败”。

## Testing

至少验证：

1. 浏览器能连接串口并发送命令。
2. 点击 `Grade 6` 后能看到等待、运行、完成状态变化。
3. 倒计时启动能自动触发。
4. 绝对时间启动能基于电脑时间自动触发。
5. 停止和取消定时能生效。
6. 日志区能持续看到设备输出。
7. 设备断开后 GUI 能正确回退到未连接状态。

## Implementation Direction

推荐分两步实现：

### Step 1

先做 GUI 框架和串口控制链路。

包含：

1. 本地网页。
2. Web Serial 连接。
3. 时间同步。
4. 倒计时和绝对时间启动。
5. 状态显示和日志区。

### Step 2

再做更细的档位和频率控制协议。

包含：

1. 电脑直接控制档位。
2. 自定义频率参数。
3. 结构化状态回显。

这样可以先把“难对时”这个核心问题解决掉，再把旋钮彻底淡出主流程。
